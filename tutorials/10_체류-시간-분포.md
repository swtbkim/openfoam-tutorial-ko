## 1. 배경

이 예제에서는 T자형 배관 내부의 유동에 대한 체류 시간 분포(Residence Time Distribution, RTD) 해석을 수행한다.

### 1.1 체류 시간 분포 (RTD)

체류 시간 분포는 반응기나 컬럼과 같은 공정 단위 내에서 Tracer 입자가 머무르는 시간을 나타내는 확률 분포 함수이다. 실제 공정에서는 이상적인 혼합이 일어나지 않는 경우가 대부분이기 때문에 화학공학적 관점에서 RTD 해석은 매우 중요하다. 예를 들어, 연속 교반 반응기(continuously stirred reactor) 내부의 실제 혼합 특성을 분석할 때 RTD를 사용한다. 그 외에도 공정 단위의 유동 패턴, 역혼합(back mixing), 단락(bypassing) 등의 정보를 얻기 위해서 RTD 해석을 활용한다.

### 1.2 Tracer 해석

![Tracer analysis](../figures/10_tracer.jpeg)

공정 단위의 RTD를 결정하기 위해서는 일반적으로 방사성 Tracer를 사용한다. Tracer를 입구에 주입한 후, 출구에서의 Tracer 농도 $C_T(t)$를 일정한 시간 간격으로 측정한다. 이렇게 측정한 농도로부터 출구의 체류 시간 분포 $E(t)$를 계산할 수 있다.

$$E(t) = \frac{C_T(t)}{\int_0^\infty C_T(t)\,dt} = \frac{\text{시간 } t \text{에서의 Tracer 농도}}{\text{전체 Tracer 농도}}$$

위 식으로부터, 시간 구간 $t$와 $t+dt$ 사이에 공정 단위를 빠져나오는 Tracer 입자의 분율은 $E(t)\,dt$임을 알 수 있다. 모든 Tracer 입자는 결국 공정 단위를 빠져나오므로, RTD는 아래와 같이 정규화된다.

$$\int_0^\infty E(t)\,dt = 1$$

### 1.3 두 솔버 연동 방식

이 예제의 가장 큰 특징은 **두 개의 솔버를 순차적으로 사용**한다는 점이다.

| 단계 | 솔버 | 풀이 대상 | 시간 의존성 |
| ------ | ------ | ----------- | ------------- |
| 1단계 | `simpleFoam` | 속도 $\mathbf{U}$, 압력 $p$, 난류량 $k$, $\varepsilon$ | 정상상태(steady) |
| 2단계 | `scalarTransportFoam` | 스칼라 농도 $T$ | 비정상(transient) |

#### 왜 두 솔버를 나누어 쓰는가?

RTD를 측정하려면 입구에 Tracer를 주입한 후 출구에서 농도가 시간에 따라 어떻게 변하는지 추적해야 한다. 즉 본질적으로 **비정상 해석**이 필요하다. 그러나 주입된 Tracer 농도가 매우 작아서 유동에 영향을 주지 않는다고 가정하면 유동장 자체는 시간에 따라 변하지 않는 **정상상태**로 가정할 수 있다.

이러한 상황에서는 운동량 방정식과 스칼라 수송 방정식을 같은 비정상 솔버로 동시에 푸는 것이 다음과 같은 이유로 비효율적이다.

1. **계산 비용**: 이미 정상상태에 도달할 유동을 매 시간 단계마다 다시 푸는 것은 낭비이다.
2. **수렴 안정성**: 유동장 수렴은 정상상태 알고리즘(SIMPLE)이 비정상 알고리즘보다 안정적이다.
3. **모듈성**: 유동장은 한 번 풀어두고, 입구 조건만 바꿔가며 RTD 계산을 여러 번 반복할 수 있다.

따라서 다음과 같은 **"동결 유동(frozen flow)" 방식**을 사용한다.

```text
 [1단계 simpleFoam]                  [2단계 scalarTransportFoam]
정상상태 유동장 U 계산        ───>      U를 그대로 가져와 동결시키고
                                       그 위에서 T만 시간 적분
```

OpenFOAM에서는 이 두 단계를 **서로 다른 케이스 디렉토리**에서 수행한다. 두 솔버가 요구하는 디렉토리 구성(특히 system/fvSchemes, system/fvSolution)이 다르기 때문이다. 1단계의 격자(blockMeshDict)와 해석 결과(U)를 2단계 케이스로 복사하는 것이 두 단계를 잇는 연결고리이다.

## 2. 예제 개요

### 예제 구성

- `simpleFoam`과 `scalarTransportFoam`을 이용하여 정사각 단면을 가지는 T자형 배관 내부의 유동을 해석하고, 두 입구에 대해 단계 함수(step function) 주입 방식으로 RTD를 계산한다.
- 입구 및 출구 단면적: 1 x 1 $\mathrm{m^2}$
- 작동 유체: 상온 대기 조건의 공기
- 작동 압력: 대기압
- Inlet 1 유속: 0.1 m/s
- Inlet 2 유속: 0.2 m/s

### 목표

- OpenFOAM을 이용한 RTD 계산 절차를 이해한다.
- 한 케이스에 두 개 이상의 솔버를 순차적으로 사용하는 방법을 이해한다.

### 데이터 처리

- 단계 응답 함수(step response function)와 RTD 곡선을 도출하여 비교한다.

## 3. 해석

### 전체 작업 흐름

이 예제는 **두 개의 케이스 디렉토리**를 사용한다. 혼동을 피하기 위해 디렉토리 이름을 미리 정해 둔다(이름은 자유롭게 바꿔도 무방하다).

- TJunctionFlow/ : `simpleFoam`으로 유동장을 계산하는 1단계 케이스
- TJunctionRTD/  : `scalarTransportFoam`으로 RTD를 계산하는 2단계 케이스

전체 작업은 다음 5단계로 진행된다.

```text
[1] TJunctionFlow 준비   (3.1)
    -> pitzDaily 튜토리얼 복사 
    -> blockMeshDict, 경계조건, 난류 모델 수정

[2] TJunctionFlow 실행   (3.2)
    -> blockMesh + simpleFoam
    -> TJunctionFlow/<lastTime>/U  (수렴된 정상상태 속도장)

[3] TJunctionRTD 준비    (3.3.1 ~ 3.3.5)
    -> scalarTransportFoam 튜토리얼 복사
    -> TJunctionFlow에서 blockMeshDict와 U 가져오기
    -> 0/T 경계조건과 controlDict 수정

[4] TJunctionRTD 실행    (3.3.6)
    -> blockMesh + scalarTransportFoam
    -> 시간에 따른 T 필드 출력

[5] 후처리               (3.4)
    -> foamToVTK + ParaView
    -> 출구 평균 T(t)
    -> RTD
```

#### 두 케이스 사이의 파일 흐름

```text
TJunctionFlow/                              TJunctionRTD/
├─ 0/                                       ├─ 0/
│  ├─ U, p, k, epsilon, ...                 │  ├─ U  <-- TJunctionFlow/<lastTime>/U 복사
├─ constant/                                │  └─ T  <-- 직접 수정
│  └─ turbulenceProperties                  ├─ constant/
├─ system/                                  │  └─ transportProperties (DT 정의)
│  ├─ blockMeshDict ─────────────────┐      ├─ system/
│  ├─ controlDict                    └──────┼─→ blockMeshDict (복사)
│  ├─ fvSchemes, fvSolution                 │  ├─ controlDict (endTime, deltaT 수정)
│  │                                        │  ├─ fvSchemes, fvSolution (그대로)
└─ <lastTime>/  (예: 1000/)                 │
   └─ U ──────────────────────────────────→ 0/U
```

이제 단계별로 진행한다.

### 3.1 1단계 케이스(TJunctionFlow) 준비

#### 3.1.1 예제 복사

OpenFOAM 튜토리얼의 simpleFoam pitzDaily 케이스를 작업 디렉토리에 복사하고 이름을 TJunctionFlow로 바꾼다.

```bash
cd $FOAM_RUN          # 작업 디렉토리로 이동
cp -r $FOAM_TUTORIALS/incompressible/simpleFoam/pitzDaily TJunctionFlow
cd TJunctionFlow
```

- system/blockMeshDict (3.1.2)
- 0/ 디렉토리의 경계 조건 파일들 (3.1.3)
- constant/turbulenceProperties (3.1.4)

#### 3.1.3 0/ 디렉토리

새로운 경계 조건(`inlet_one`, `inlet_two`, `outlet`, `walls`)에 맞게 p, U, nut, nuTilda, k, epsilon 파일을 모두 수정한다. 예를 들어, U 파일은 아래와 같이 수정한다.

```cpp
// * * * * * * * * * * * * * * * * * * * * * * * * * * * * *//
dimensions      [0 1 -1 0 0 0 0];

internalField   uniform (0 0 0);

boundaryField
{
    inlet_one
    {
        type            fixedValue;
        value           uniform (0.1 0 0);
    }
    inlet_two
    {
        type            fixedValue;
        value           uniform (-0.2 0 0);
    }
    outlet
    {
        type            zeroGradient;
    }
    walls
    {
        type            fixedValue;
        value           uniform (0 0 0);
    }
}
// * * * * * * * * * * * * * * * * * * * * * * * * * * * * *//
```

> **Note:** `inlet_two`의 속도 $x$성분이 음수(-0.2)인 이유는 이 입구가 도메인의 양의 $x$방향 끝에 위치하므로, 도메인 안쪽으로 들어오는 방향이 $-x$이기 때문이다.

> **Note:** 난류 파라미터는 모두 유속 10 m/s를 기준으로 설정된 값이므로, 적절히 스케일링하여 사용한다.

#### 3.1.4 constant/ 디렉토리

turbulenceProperties 파일에서 난류 모델을 `kEpsilon`으로 설정한다.

### 3.3 2단계 케이스(TJunctionRTD) 준비 및 실행 (`scalarTransportFoam`)

#### 3.3.1 예제 복사

`scalarTransportFoam` 솔버 설정을 위해 별도의 디렉토리(TJunctionRTD/)로 튜토리얼 케이스를 복사한다.

```bash
cd $FOAM_RUN          # TJunctionFlow가 있는 동일한 작업 디렉토리
cp -r $FOAM_TUTORIALS/basic/scalarTransportFoam/pitzDaily TJunctionRTD
```

이제 작업 디렉토리에는 두 케이스가 나란히 존재한다.

```text
$FOAM_RUN/
├─ TJunctionFlow/    (1단계 simpleFoam 케이스)
└─ TJunctionRTD/     (2단계 scalarTransportFoam 케이스)
```

이후의 모든 파일 수정은 TJunctionRTD/ 안에서 이루어진다.

#### 3.3.2 격자 동기화: system/blockMeshDict 복사

TJunctionRTD/에 들어 있는 원래 격자는 pitzDaily 형상이므로, 1단계에서 사용한 T자형 격자로 교체해야 한다. **두 케이스의 격자가 정확히 동일해야 1단계의 U 필드를 셀 단위로 그대로 사용할 수 있다.**

```bash
cp TJunctionFlow/system/blockMeshDict TJunctionRTD/system/blockMeshDict
```

이 격자가 다르면 `scalarTransportFoam` 실행 시 셀 개수 불일치로 즉시 오류가 발생한다.

#### 3.3.3 속도장 가져오기: 0/U 복사

이것이 두 솔버를 잇는 **가장 핵심적인 작업**이다.

TJunctionFlow/에서 simpleFoam이 수렴한 마지막 시간 디렉토리(예: 270/)에 들어 있는 U 파일을 TJunctionRTD/0/U로 덮어쓴다.

```bash
# (1) 마지막 시간 디렉토리 이름 확인
ls TJunctionFlow/
# 0  100  200  300  ...  1000  constant  system

# (2) 마지막 시간 디렉토리(여기서는 1000)의 U 파일 복사
cp TJunctionFlow/1000/U TJunctionRTD/0/U
```

> **Note:** TJunctionFlow/0/U가 아니라 TJunctionFlow/270/U(또는 자신의 마지막 시간)를 복사해야 한다.
>
> - TJunctionFlow/0/U : 시뮬레이션 시작 시 초기값 (수렴 전)
> - TJunctionFlow/270/U : 수렴된 정상상태 속도장

이렇게 복사된 TJunctionRTD/0/U는 더 이상 수정하지 않는다. `scalarTransportFoam`은 이 속도장을 시간 적분 내내 **동결**(frozen)시켜 사용한다.

> **Note: U 외의 다른 변수는 어떻게 되는가?**
>
> simpleFoam은 $p$, $k$, $\varepsilon$, $\nu_t$ 등도 함께 계산하지만, `scalarTransportFoam`은 스칼라 수송 방정식($\partial T/\partial t + \nabla \cdot (\mathbf{U}T) = \nabla \cdot (D_T \nabla T)$)만 풀기 때문에 $\mathbf{U}$ 외의 변수는 전혀 사용하지 않는다. 따라서 U만 복사하면 충분하다.

#### 3.3.4 Tracer 정의: 0/T 수정

T는 우리가 추적할 비반응성 스칼라(inert scalar)이다. **inlet_one에 대한 RTD를 계산하려면** TJunctionRTD/0/T 파일을 다음과 같이 수정한다.

```cpp
// TJunctionRTD/0/T
// * * * * * * * * * * * * * * * * * * * * * * * * * * * * *//
dimensions      [0 0 0 0 0 0 0];    // Dimensionless scalar

internalField   uniform 0;          // Tracer concentration = 0 at t=0

boundaryField
{
    inlet_one
    {
        type            fixedValue;
        value           uniform 1.0;   // 단계 함수 입력: 농도 1.0으로 계속 주입
    }
    inlet_two
    {
        type            fixedValue;
        value           uniform 0;     // 비교 대상 입구는 Tracer 없음
    }
    outlet
    {
        type            zeroGradient;  // 출구는 자유 유출
    }
    walls
    {
        type            zeroGradient;  // 벽에서 Tracer 출입 없음
    }
}
// * * * * * * * * * * * * * * * * * * * * * * * * * * * * *//
```

물리적 의미: $t=0$에서 inlet_one에 농도 1.0짜리 Tracer가 정상 유동을 따라 들어오기 시작하고, 그것이 출구에 도달하여 출구 농도가 시간에 따라 변하는 양상을 보는 것이 **단계 응답**(step response)이다.

#### 3.3.5 시간 설정: system/controlDict 수정

TJunctionRTD/system/controlDict에서 다음 항목을 수정한다.

| 항목 | 원래 값 | 수정값 | 이유 |
| ------ | -------- | -------- | ------ |
| `endTime` | 0.1 | **120** | 이상 체류 시간(약 60초)의 약 2배 |
| `deltaT` | 0.0001 | **0.1** | Courant 수 $\mathrm{Co} \approx 0.4$가 되도록 |

> **권장사항(필수 아님)**: 다음 항목들도 함께 정리해 두면 후처리가 편하다.
>
> | 항목 | 권장값 | 효과 |
> | ------ | -------- | ------ |
> | `writeControl` | `runTime` | 시간 단위 간격으로 결과 파일 출력 |
> | `writeInterval` | `1` | 1초 간격으로 출력 (총 약 120개 시간 디렉토리) |
> | `purgeWrite` | `0` | 모든 시간 단계 결과 보존 |
>
> 만약 `writeInterval`을 너무 작게(예: 0.1) 두면 결과 파일이 1200개 이상 생성되어 후처리가 무거워진다.

#### 3.3.6 시뮬레이션 실행

이제 TJunctionRTD/에서 격자를 생성하고 솔버를 실행한다.

```bash
cd TJunctionRTD
blockMesh
scalarTransportFoam
foamToVTK             # ParaView 후처리용 VTK 변환
```

`scalarTransportFoam`은 매 시간 단계마다 0/U(동결된 속도장)를 읽고 T 필드를 시간 적분한다. 실행이 끝나면 TJunctionRTD/ 안에 1/, 2/, ..., 120/과 같은 시간 디렉토리들이 만들어지고, 각 디렉토리에는 그 시각의 `T` 필드가 저장된다.

### 3.4 후처리

TJunctionRTD/ 디렉토리에서 `paraFoam`을 실행한다.

#### 3.4.1 Tracer 확산 전체 가시화 — Slice 필터

Slice 필터를 적용하여 T자 배관 내부의 전체적인 Tracer 분포를 확인한다. 

![inlet_one에 대해 t=120 s에서 스칼라 T의 등치선 플롯](../figures/10_T_contour.jpeg)

#### 3.4.2 출구 평균 T 계산 — Integrate Variables 필터

1. **Mesh Regions**에서 `patch/outlet`만 체크하고 다른 patch는 모두 체크 해제
2. **Cell Arrays**에서 `T`만 체크 (`U`, `p`는 RTD 계산에 불필요)
3. `Apply`

이제 RenderView에는 1 m × 1 m 의 출구 단면 한 장만 표시된다. 시간 슬라이더를 옮기면 그 단면 위의 T 분포가 시간에 따라 변하는 것이 보인다.

선택된 출구 patch가 활성 노드인 상태에서 다음 메뉴를 선택한다.

```text
Filters > Data Analysis > Integrate Variables
```

(필터 이름이 잘 안 보이면 단축키 `Ctrl+Space`로 검색해도 된다.) `Apply`를 누르면 Pipeline Browser에 `IntegrateVariables1` 노드가 추가된다. 이 필터는 입력 표면의 모든 셀에 대해 변수를 면적 가중 적분한 단일 셀 데이터셋을 출력한다.

결과를 표로 보려면 RenderView에서 새 레이아웃을 열고 `SpreadSheet View`를 클릭한다. IntegrateVariables1이 선택된 상태에서 SpreadSheetView 상단의 **Attribute** 드롭다운을 `Cell Data`로 둔다. 표에는 다음 컬럼이 보인다.

- **T**: 출구 면에 대한 $\int T\, dA$
- **Area** (필드 데이터 탭에서 확인): 1.0

이 케이스의 출구 면적은 정확히 $1\,\mathrm{m}^2$이므로 `T` 컬럼의 값이 곧 **출구 평균 T**이다. 면적이 1이 아닌 케이스에서는 `Filters > Common > Calculator`로 `T/Area` 식을 만들어 명시적으로 평균을 정의하는 것이 안전하다.

VCR로 시간 슬라이더를 옮기면 평균 T 값이 0 → 점진 증가 → 정상상태값으로 변하는 것이 표에서 보인다.

#### 3.4.3 시간에 따른 평균 T(t) — 단계 응답 곡선

지금까지의 IntegrateVariables는 **한 시간 단계의 적분값**만 표시한다. 모든 시간 단계의 평균 T를 한꺼번에 곡선으로 그리려면 다음 필터를 사용한다.

```text
Filters > Data Analysis > Plot Data Over Time
```

> **Note:** 비슷한 이름의 `Plot Selection Over Time`은 **사용자가 RenderView에서 마우스로 선택한 셀/포인트**만 추적하는 다른 필터이다. IntegrateVariables 출력처럼 단일 셀 데이터셋의 시간 변화를 그릴 때는 반드시 `Plot Data Over Time`을 사용해야 한다.

1. Pipeline Browser에서 `IntegrateVariables1` 선택
2. `Filters > Data Analysis > Plot Data Over Time`
3. Properties 패널의 `Field Association`은 `Cells` 그대로 둠
4. `Apply`

`PlotDataOverTime1` 노드가 추가되면서 Line Chart View가 자동으로 열린다. 가로축은 시간(0~120 s), 세로축은 출구 평균 T이다. 이것이 **단계 응답 함수** $F(t)$이다.

차트 좌측 범례의 선 항목 중 `T`만 남기고 나머지(`Area`, `vtkOriginalIndices` 등)는 체크를 해제하면 가독성이 좋아진다.

![두 입구에 대한 출구의 평균 T 값의 시간 변화](../figures/10_step_response.jpeg)

시간이 충분히 지나면 평균 T는 상수 값에 수렴한다. 이 수렴값은 활성화한 입구의 부피 유량 비율과 같다.

- inlet_one 활성화 시: $T_\infty = Q_1 / (Q_1 + Q_2) = 0.1/(0.1+0.2) \approx 0.333$
- inlet_two 활성화 시: $T_\infty = Q_2 / (Q_1 + Q_2) = 0.2/(0.1+0.2) \approx 0.667$

수렴값이 위 이론값에 가깝게 도달하는지 확인하는 것이 1차 검증이다.

#### 3.4.4 inlet_two 케이스 추가 실행

지금까지의 단계 응답은 inlet_one에 농도 1.0을 주입한 경우의 결과이다. RTD 비교를 위해 inlet_two에 대해서도 동일한 절차를 반복한다. 수정해야 할 부분은 **0/T의 경계 조건뿐**이고, 격자·속도장·시간 설정은 모두 동일하다.

```cpp
// TJunctionRTD/0/T (inlet_two 버전)
boundaryField
{
    inlet_one
    {
        type            fixedValue;
        value           uniform 0;     // 0으로 변경
    }
    inlet_two
    {
        type            fixedValue;
        value           uniform 1.0;   // 1.0으로 변경
    }
    outlet
    {
        type            zeroGradient;
    }
    walls
    {
        type            zeroGradient;
    }
}
```

inlet_one 결과를 보존한 채로 inlet_two 계산을 수행하려면 두 가지 방법이 있다.

**옵션 1**: 케이스를 별도로 복사하여 각각 보관

```bash
cp -r TJunctionRTD TJunctionRTD_inlet1   # inlet_one 결과 백업
cp -r TJunctionRTD TJunctionRTD_inlet2   # inlet_two용 새 케이스
# 이후 TJunctionRTD_inlet2/0/T를 inlet_two 버전으로 수정 후 재실행
```

**옵션 2**: 같은 케이스에서 시간 디렉토리만 정리 후 재실행

```bash
cd TJunctionRTD
foamListTimes -rm     # 0을 제외한 모든 시간 디렉토리 삭제
# 이후 0/T를 inlet_two 버전으로 수정 후 다시 scalarTransportFoam 실행
```

inlet_two 계산이 끝나면 3.4.1 ~ 3.4.3의 절차를 반복하여 그 케이스의 단계 응답 곡선도 ParaView로 추출한다.

#### 3.4.5 데이터 CSV로 내보내기

각 inlet 케이스에서 곡선 데이터를 외부 도구로 가져가기 위해 `PlotDataOverTime1` 노드를 선택한 상태에서

```text
File > Save Data ...
```

확장자를 `.csv`로 지정하고 두 케이스 각각을 저장한다 (예: `step_response_inlet1.csv`, `step_response_inlet2.csv`). 시간 컬럼과 평균 T 컬럼이 포함된다.

![두 입구에 대한 RTD 곡선](../figures/10_RTD.jpeg)

inlet_two는 inlet_one보다 유속이 2배(0.2 vs 0.1 m/s)이므로 평균 체류 시간 $\tau$가 더 짧고 peak가 더 일찍 나타난다. 두 RTD 곡선의 모양 차이는 T자 배관에서 두 입구가 점유하는 유로의 길이와 속도장 구조를 반영한다.
